Introduction

A Retrieval-Augmented Generation (RAG) system is only as good as what it retrieves. If the wrong chunks reach the language model, no amount of prompt engineering will produce a correct, well-cited answer.
While building a local-first RAG system for PDF documents (Docling, parsing, structure-aware chunking), I needed a way to measure whether the retrieval side of the pipeline was actually working, without hand-labelling questions for every document I ingested.

This post describes the evaluation pipeline I built on top of the existing system: an automatic Q&A dataset generator that mines the vector store itself, and a scorer that turns that dataset into Recall@K and MRR numbers.

Why evaluate retrieval at all

In the previous article of this series, we built a simple Retrieval-Augmented Generation (RAG) pipeline and focused on the essential components required to retrieve relevant context for an LLM. The system works, but before adding new features, it is worth answering a simple question: how well does the current retrieval pipeline actually perform?

This article introduces an evaluation framework that allows us to answer that question in a quantitative way. We will use it to measure the performance of the RAG developed in the first part of the series, establishing a baseline against which future improvements can be compared.

As the series progresses, we will extend the original architecture with more advanced retrieval techniques. In the next article, for example, we will implement a hybrid search strategy that combines semantic and keyword-based retrieval, and later we will explore additional optimizations to further improve the quality of the retrieved results.

Without a consistent evaluation process, determining whether these changes are beneficial would be largely subjective. A new component might appear to work well for a few examples while actually reducing the overall retrieval quality across the dataset.

The goal of the evaluation pipeline is to replace intuition with measurable metrics. By tracking values such as Recall@K and Mean Reciprocal Rank (MRR), we can objectively compare different versions of the system and quantify the impact of each enhancement introduced throughout this series.

Another important design goal is to avoid the need for manually labelled datasets. Instead, the evaluation set is generated automatically from the documents that are already indexed, leveraging the same local LLM infrastructure used by the RAG system itself.

Building the evaluation dataset

The dataset generator, EvalDatasetGenerator, lives in evaluation/eval_dataset_generator.py and exposes a single entry point: 

generator = EvalDatasetGenerator(vector_store, ollama_client)
summary = generator.generate(output_path=EVAL_DATASET_PATH)
# -> {"total_questions": N, "cluster_k": K, "output_path": "..."}

Internally, generate() runs through five steps:

  1. Fetch all embeddings stored in Milvus.
  2. Cluster the embeddings to group similar chunks together.
  3. Sample a set of seed chunks from the clustered data.
  4. Build neighbourhood groups around each seed chunk based on similarity.
  5. Use the LLM to transform each group into question/answer pairs.

Each step is described in more detail below.



Fetch all embeddings stored in Milvus

The first step calls vector_store.fetch_all_embeddings(), which pages through every entity in the academic_chunks collection using the same pagination pattern as fetch_all_content() and returns, for every chunk, its chunk_id, paper_id and 768-dimensional embedding. These are assembled into a single (N, 768) numpy array.

If there isn't enough data — fewer than EVAL_CLUSTER_K_MIN * EVAL_CHUNKS_PER_GROUP chunks — generation aborts early with a ValueError asking for more documents to be ingested first.

Clustering the embedding space

The embeddings are then clustered with sklearn.cluster.KMeans. Rather than picking a fixed number of clusters, the number K is chosen automatically with a silhouette-score sweep:

  • An initial heuristic gives a starting point: $$k_{init} = \max\big(K_{min}, \min(K_{max}, \lfloor\sqrt{N/2}\rfloor)\big)$$
  • For every candidate K in [EVAL_CLUSTER_K_MIN, min(EVAL_CLUSTER_K_MAX, N // EVAL_CHUNKS_PER_GROUP)], a KMeans(n_clusters=K, n_init=10, random_state=42) is fit and scored with silhouette_score. When N exceeds 5000, the score is computed on a random subsample of 5000 points to keep the sweep fast — the final model is still fit on all N embeddings.
  • The silhouette score for a point i compares its average distance to points in its own cluster, a(i), against its average distance to the nearest other cluster, b(i): $$s(i) = \frac{b(i) - a(i)}{\max(a(i), b(i))}$$ The K with the highest mean silhouette score is kept, and a final KMeans is fit on the full embedding set with that K, producing one cluster label per chunk.

Stratified seed sampling

With the embedding space clustered, the next question is: which chunks become the "seeds" around which evaluation questions are built?

The target number of seeds is n_seeds = ceil(EVAL_NUM_QUESTIONS / EVAL_QUESTIONS_PER_GROUP) — with the default configuration (50 questions, 3 per group) that is 17 seeds.

Seeds are sampled stratified by document, not by cluster: each document's share of seed slots is proportional to its share of total chunks, with the rounding adjusted so the totals add up to n_seeds exactly. Within a document, slots are filled by sampling chunk indices uniformly at random (without replacement if the document has enough chunks, with replacement otherwise).

This matters because without stratification, a handful of very large documents would dominate the evaluation set and small documents would never be tested at all.

Assembling neighbour groups

Each seed becomes the centre of a small "group" of related chunks that are shown to the LLM together, so that some of the generated questions can require connecting information across chunks.

For a given seed:

  • All other chunks sharing the seed's cluster label are candidates.
  • Because the embedder L2-normalises every vector (normalize_embeddings=True), cosine similarity between two embeddings reduces to a plain dot product: $$\text{sim}(a, b) = a \cdot b = \cos(\theta)$$ This is computed in-memory with numpy against every same-cluster embedding.
  • The top EVAL_CHUNKS_PER_GROUP - 1 chunks by similarity are taken as neighbours (default: 3 neighbours + 1 seed = 4 chunks per group), and the group is de-duplicated by chunk_id.

The full Chunk objects for the group — including content, title, year, section and page — are then materialised in one call to vector_store.fetch_by_ids(group_chunk_ids).

Generating Q&A pairs with the LLM

Each group of chunks is formatted into a prompt that lists every passage with its chunk_id, document title, year and section, and asks ollama_client.generate() for exactly EVAL_QUESTIONS_PER_GROUP question/answer pairs as a JSON array — no markdown, no extra text.

The prompt explicitly asks for a mix of difficulties:

  • easy — single-passage factual recall
  • medium — requires connecting two passages
  • hard — requires inference, comparison, or critical thinking
and, crucially, for each pair the model must also return the relevant_chunk_ids from the group that are needed to answer it — this becomes the ground truth used for scoring in last phase.

The response is parsed with json.loads(). If parsing fails, or an item is missing one of the four required keys (question, answer, difficulty, relevant_chunk_ids), the problem is logged with logging.warning and that item — or the whole group — is skipped rather than raising. Groups are processed until total_collected >= EVAL_NUM_QUESTIONS, then generation stops early.

Every accepted item is assigned a question_id (UUID4) plus cluster_id and source_paper_ids derived from its group, and the whole collection is written to data/eval_dataset.json alongside a generation timestamp, the model name, the total question count and the chosen K.

Scoring retrieval quality

The evaluation loop is implemented in evaluation/eval_scorer.py. The EvalScorer(pipeline) class loads data/eval_dataset.json and, for each item, calls pipeline.retrieve(question, top_k) — the retrieval-only path of the system that returns the top candidate chunks without generating a final answer.

For every question, the retrieved chunk_ids are compared against the relevant_chunk_ids defined in the dataset. This comparison is used to compute standard information retrieval metrics at K = 1, 3, and 5.

  • Recall@K — the fraction of relevant chunks that appear anywhere in the top K retrieved results: $$\text{Recall@K} = \frac{|\{\text{relevant chunks}\} \cap \{\text{retrieved}_{1..K}\}|} {|\{\text{relevant chunks}\}|}$$
  • MRR (Mean Reciprocal Rank) — rewards ranking the first relevant chunk as high as possible: $$MRR = \frac{1}{|Q|}\sum_{i=1}^{|Q|} \frac{1}{\text{rank}_i}$$ where ranki is the position of the first relevant chunk for question i (or the term is 0 if no relevant chunk was retrieved).

The scorer writes per-question scores plus an aggregate, broken down by the easy / medium / hard difficulty labels generated in previous steps, to data/eval_results.json, and prints a summary table to stdout.

Because the difficulty breakdown comes for free from previous steps, a configuration change that only hurts "hard" questions (e.g. ones that need cross-passage reasoning) is visible immediately, even if the aggregate Recall@5 barely moves.

Configuration and CLI

Every hyperparameter for both phases lives in config.py, alongside the rest of the pipeline's settings: 

EVAL_DATASET_PATH        = BASE_DIR / "data" / "eval_dataset.json"
EVAL_RESULTS_PATH        = BASE_DIR / "data" / "eval_results.json"
EVAL_NUM_QUESTIONS       = 50    # total Q&A pairs to generate
EVAL_CLUSTER_K_MIN       = 5     # min K for silhouette-score sweep
EVAL_CLUSTER_K_MAX       = 30    # max K for silhouette-score sweep
EVAL_CHUNKS_PER_GROUP    = 4     # chunks per LLM call (1 seed + 3 neighbours)
EVAL_QUESTIONS_PER_GROUP = 3     # Q&A pairs generated per chunk group
EVAL_LLM_MODEL           = OLLAMA_CHAT_MODEL

Both phases are wired into main.py as standalone CLI flags, so they can be run independently of the interactive chat: 
# mine the vector store and generate eval_dataset.json
python main.py --eval-generate

# run retrieval-only queries and score them
python main.py --eval-score
--eval-generate only needs the vector store and the Ollama client (no full pipeline startup), prints the chosen cluster K and silhouette score, and writes eval_dataset.json. --eval-score requires that file to already exist, runs pipeline.retrieve() for every question, and writes eval_results.json with the Recall@K / MRR breakdown.



Baseline results

To establish a concrete baseline, we ran the full evaluation pipeline against a small but diverse corpus of three books:

  • Hormonal by Martie Haselton — on the evolutionary role of hormones in women's behaviour.
  • The Mind-Gut Connection by Emeran Mayer — on gut-brain communication and the microbiome.
  • The Metabolism Reset by Lara Briden — on metabolic health, nutrition, and hormonal balance.

The retrieval method used for this baseline is plain vector search — cosine similarity over the Milvus index with no hybrid search component and no reranker. This gives us a clean starting point: any improvement introduced in later articles (BM25 fusion, cross-encoder reranking, query expansion) can be measured directly against these numbers.


Test corpus

The silhouette-score sweep selected K = 5 clusters, and the generator produced 50 question–answer pairs distributed across the three books. The difficulty split was roughly balanced: 17 easy, 17 medium, and 16 hard questions. Each question carries its ground-truth relevant_chunk_ids, so retrieval quality can be scored automatically without manual labelling.

The three books cover different domains — evolutionary psychology, neurogastroenterology, and clinical nutrition — which forces the evaluation to test retrieval across genuinely distinct regions of the embedding space rather than within a single homogeneous topic.

Aggregate metrics

With top_k = 5, the baseline retrieval scores are:


Metric Value
MRR 0.745
Recall@1 0.430
Recall@3 0.660
Recall@5 0.697

An important detail to keep in mind when reading these numbers: Recall@K is normalised by the number of relevant chunks for each question. Many of the generated questions — especially medium and hard ones — have two or three ground-truth relevant_chunk_ids. This means that even when retrieval works perfectly at rank 1, the score for that question is 1/2 = 0.5 or 1/3 ≈ 0.33, not 1.0. Recall@1 can only reach 1.0 for questions that depend on a single chunk. As a result, the aggregate Recall@1 of 0.430 is not as low as it might first appear — it is structurally bounded by the multi-chunk nature of the evaluation set.

With that in mind, an MRR of 0.745 means that, on average, the first relevant chunk appears between rank 1 and rank 2 — a reasonable starting point, but with clear room for improvement. Recall@5 at ~70% tells us that nearly a third of the ground-truth chunks are still missing from the top 5 results, which directly limits how much relevant context the LLM receives during generation.

Breakdown by difficulty

The per-difficulty breakdown reveals where plain vector search struggles most:


Difficulty n MRR Recall@1 Recall@3 Recall@5
Easy 17 0.578 0.471 0.647 0.647
Medium 17 0.887 0.539 0.696 0.775
Hard 16 0.771 0.271 0.635 0.667

A few patterns stand out. Medium questions achieve the highest MRR (0.887) and the best Recall@5 (0.775), outperforming both easy and hard questions. This is likely because medium questions require connecting two passages that tend to be semantically close — the kind of relationship that cosine similarity handles well.

Easy questions, despite targeting single-passage factual recall, show a surprisingly low MRR of 0.578. Inspecting the per-question results reveals that several easy questions scored zero across all metrics — the relevant chunk was simply not retrieved in the top 5. This happens when the question's phrasing diverges enough from the chunk's vocabulary that pure semantic similarity fails to bridge the gap, a classic weakness of embedding-only retrieval that keyword-based methods can address.

Hard questions have the lowest Recall@1 (0.271), which is expected: these questions require cross-passage inference, so the single top result is rarely sufficient. However, Recall@3 recovers to 0.635, suggesting that relevant chunks do appear in the result set — just not at rank 1.

The fact that easy Recall@5 plateaus at 0.647 (identical to Recall@3) while medium Recall@5 climbs to 0.775 confirms that the missing easy chunks sit outside the top-K window entirely. A reranker or hybrid search strategy should help surface these chunks by complementing semantic similarity with lexical matching — a direction we will explore in the next article.

Conclusion

The result is a small, self-contained loop: we cluster what is already indexed, sample it in a way that avoids over-representing large documents, and then use a local LLM to generate grounded questions with their corresponding ground-truth chunk IDs. These questions are then replayed through the actual retrieval pipeline to compute standard metrics such as Recall@K and MRR.

Because the dataset includes both questions and reference answers, this setup also opens the door to a more advanced evaluation layer. We can introduce an LLM-as-a-judge approach to assess the correctness and quality of generated answers, and even extend it to verify whether specific guardrails or constraints are being respected during generation.

In one of the next articles, we will explore how to implement this judging mechanism in practice, and how to integrate MLflow into the pipeline to track experiments, evaluation metrics, and system variations in a structured way.



None of this requires a human to read a single document or write a single question by hand, which means the benchmark can be regenerated whenever the document set changes, and re-scored instantly whenever a retrieval setting in config.py changes — turning "I think the new fusion weights are better" into a number that can be compared run over run.
