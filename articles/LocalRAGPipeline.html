Introduction

Most explanations of Retrieval-Augmented Generation (RAG) start from the language model and treat retrieval as a black box that magically "fetches relevant context". In practice, almost all of the engineering effort — and almost all of the failure modes — live inside that black box.

This post walks through the first, simplest version of a local RAG system I built to chat with a folder of PDF documents: a pipeline that parses PDFs into structured documents, chunks them into retrievable units, embeds those chunks into vectors, stores them in a vector database, and retrieves the most relevant ones by plain cosine similarity before handing them to a language model.

Everything described here runs entirely on a local machine: no proprietary APIs, no cloud calls, no API keys. A follow-up post builds on top of this "vanilla" pipeline to add BM25 sparse retrieval and a cross-encoder reranker — the parts that turn a toy demo into something that holds up against real questions.

From PDFs to structured documents

A PDF is a description of where ink goes on a page, not a document. Before anything else can happen — chunking, citing a page number, telling a table apart from a paragraph — the raw PDF has to become a structured representation: an ordered list of typed elements (titles, section headers, paragraphs, tables, figures, formulas), each tagged with the page it came from.

This pipeline uses Docling, an open-source document-conversion library, to do that heavy lifting. A DocumentConverter reads the PDF and exposes doc.iterate_items(), a stream of typed items — TextItem, TableItem, PictureItem — each labelled with a DocItemLabel (TITLE, SECTION_HEADER, TEXT, TABLE, PICTURE, FORMULA, …).

From this stream the parser builds a single ParsedDocument:

  • title — taken from the first TITLE item
  • year — inferred with a regex, first from the filename, then from the first 2000 characters of text
  • authors — a best-effort heuristic: text blocks between the title and the first section header on page 1/2 are accepted as "author-like" if they have at least two capitalised words, no sentence-ending punctuation, and no affiliation keywords (university, department, @, …). It works well for standard arXiv-style PDFs and is skipped silently otherwise
  • elements — the ordered list of RawElements, each carrying its label, text (or markdown for tables, caption for figures), page and heading level

Tables are exported with item.export_to_markdown(doc) so that their row/column structure is preserved as plain text, while figures currently retain only their captions. At this stage, there is no OCR or image captioning, meaning that a figure without a caption or relevant surrounding text is discarded during the chunking process.

In a future upgrade, this limitation will be addressed by integrating MinerU into the document-processing pipeline. MinerU provides high-quality document parsing with OCR, layout analysis, and extraction of images, tables, formulas, and other structured elements, enabling a more complete and faithful representation of the original document and supporting a robust pipeline for correctly handling all document components.

One practical note: on first use, Docling downloads its own internal ML models for layout detection and OCR. This takes a few minutes once and is cached afterwards — everything still runs locally, just with a one-time setup cost.

Chunking: cutting documents into retrievable units

An embedding model has a context limit, and even if it didn't, embedding an entire 12-page paper as a single vector would be useless — the resulting vector would be an average of dozens of unrelated ideas, similar to everything and useful for nothing. Retrieval needs chunks: pieces of text small enough to be coherent, large enough to carry meaning on their own.

The chunker walks the ordered elements list and applies a few simple rules:

  • Section headers flush whatever text has been accumulated into a chunk, then update the "current section" label used to tag every following chunk
  • Text, list items and paragraphs accumulate into a buffer
  • Tables, figures and formulas become their own standalone chunks
  • Long buffers are split on word boundaries with overlap, controlled by CHUNK_MAX_TOKENS = 512 and CHUNK_OVERLAP_TOKENS = 50

Tables, figures and equations are a special case: read in isolation, a markdown table or a bare LaTeX formula embeds poorly — its meaning lives in the prose around it. So the chunker stitches in 50 words of the immediately preceding text (taken from the buffer right before it's flushed) and the next 50 words that follow (scanned ahead until the next section header), producing something like:

Markdown tables are truncated to 6 000 characters (always keeping the header row, with a "[... N rows truncated]" note), and every chunk's content is capped at 8 000 characters — the maximum length of the content column in the vector store. A figure with no caption is still kept as a "Figure (no caption)" chunk as long as there's surrounding text to stitch in; only truly isolated, caption-less figures get dropped.

Every table, figure and formula also gets a sequential ID (tbl-1, fig-2, eq-3, …) stored in its chunk's metadata, which later enables targeted queries like "show me table 3 from this paper".

The output of this stage is a flat list of Chunk objects, each carrying chunk_id, paper_id, title, authors, year, section, a chunk_type (text / table / figure / equation), the content string, and a small metadata object with page number and the relevant table/figure/equation ID.

Embeddings: turning text into vectors

To compare a question against thousands of chunks, both need to live in the same space — a space where "similar meaning" corresponds to "small distance". That's what an embedding model provides: a function that maps a piece of text to a fixed-length vector of real numbers.

This pipeline uses BAAI/bge-base-en-v1.5, a 768-dimensional retrieval-tuned model hosted on Hugging Face, run locally through sentence-transformers. The weights are downloaded once and cached — there's no API call per embedding, just a local forward pass through the model.

A few details that matter in practice:

  • This is a bi-encoder: each chunk is embedded independently, with no knowledge of the query it will eventually be compared against. This is what makes it possible to embed an entire corpus once, ahead of time, and store the vectors
  • All vectors are L2-normalised (normalize_embeddings=True), so cosine similarity between two vectors reduces to a plain dot product: $$\text{sim}(q, d) = \hat{q} \cdot \hat{d} = \cos(\theta)$$
  • BGE models are trained with an asymmetric convention: document passages are encoded as-is, but queries are prefixed with an instruction — "Represent this sentence for searching relevant passages: " — before embedding. Only the content field is ever embedded; none of the metadata (title, section, page, …) is

The result is one 768-dimensional vector per chunk, ready to be written to the vector store alongside the rest of the chunk's fields.

The vector store: Milvus

Embeddings are only useful if they can be searched quickly. With a few thousand chunks, comparing a query vector against every stored vector by brute force is still fast enough — but it doesn't scale, and a dedicated vector database brings persistence, metadata filtering and a proper index for free.

This pipeline uses Milvus in standalone mode (etcd + MinIO + milvus-standalone, via Docker Compose), accessed through the pymilvus MilvusClient API. Each chunk becomes one entity in an academic_chunks collection: 

chunk_id    VARCHAR (PK, 64)   UUID
paper_id    VARCHAR (256)      PDF filename stem
title       VARCHAR (512)      from parsing
authors     VARCHAR (512)      from parsing
year        INT64              from parsing
section     VARCHAR (512)      from parsing
chunk_type  VARCHAR (32)       text | table | figure | equation
content     VARCHAR (8192)     the chunk text
page        INT64              from parsing
embedding   FLOAT_VECTOR (768) from the embedder

The collection is indexed with HNSW (Hierarchical Navigable Small World graphs, see Malkov & Yashunin, 2016) using the COSINE metric. HNSW organises vectors into a multi-layer graph in which each vector is connected to a limited number of nearby neighbours. During a search, the algorithm starts from a sparse upper layer and greedily traverses the graph while descending to denser layers, yielding highly accurate approximate nearest-neighbour retrieval with sub-linear search complexity even for collections containing millions of vectors.



The index is configured with M=16 and efConstruction=128. The parameter M specifies the maximum number of neighbour connections that each node can maintain in the graph. Larger values generally improve recall and search quality by creating a denser graph, at the cost of increased memory usage and longer index construction times. A value of 16 is a common compromise between accuracy and efficiency.



The parameter efConstruction controls the breadth of the search performed while building the graph. Higher values allow the algorithm to consider more candidate neighbours before establishing connections, resulting in a higher-quality index and better retrieval accuracy, although construction becomes slower. The chosen value of 128 provides good index quality while keeping build times practical for large datasets.



One quirk worth calling out: insert() calls flush() immediately after every write. Without it, count() and the collection stats report 0, because Milvus only counts sealed segments — the data is there, it's just invisible until flushed.

Semantic retrieval

With everything embedded and indexed, the "vanilla" query path is short:

  • 1. Embed the user's question with the same model, using the
    "Represent this sentence for searching relevant passages: " instruction prefix
  • 2. Run an approximate nearest-neighbour search over the HNSW index for the top K chunks by cosine similarity
  • 3. Return those chunks, each paired with its similarity score

That's it — no keyword matching, no re-scoring, no filtering. The top K chunks by raw vector similarity become the candidate set that gets passed on to the language model. It works surprisingly well for conceptual, "explain X to me" style questions, because embedding models are good at capturing topical similarity.

Where it struggles — and the motivation for the next post — is anything that hinges on an exact token: a specific model name, an equation symbol, a table number, a year. Two passages can be about completely different models yet sit close together in embedding space simply because they discuss the same topic in similar language. Pure vector search has no mechanism to break that tie in favour of the passage that actually contains the term the user typed.

Closing the loop: local generation

The retrieved chunks are handed to a ChatEngine, which formats them into a numbered context block and sends it, together with the conversation history (a rolling 6-turn window) and the user's question, to a local language model via Ollama — by default llama3.1, but any model pulled into Ollama works.

The system prompt enforces two rules: answer only from the provided context, and cite sources inline in the form [paper_id, Section: …, Page: …] — the paper_id, section and page fields carried by every chunk since the parsing stage make this possible without any extra lookup.

Like the embedding model, Ollama runs entirely on the local machine. The only "calls" in this whole pipeline are to processes running on localhost.

Conclusion

Parsing with Docling, structure-aware chunking with context stitching, embedding with a BGE bi-encoder, HNSW search in Milvus, and grounded generation with a local Ollama model — together these form a complete, local-first RAG loop. For many questions, this "vanilla" pipeline is already enough.

But pure vector similarity has a blind spot: it's a single, lossy summary of meaning, with no way to guarantee that an exact word the user cared about actually shows up in what gets retrieved.

That raises an important question: how do we actually evaluate whether this system is working well? In the next post, we’ll tackle that directly — introducing evaluation strategies and metrics for RAG systems. You can find it here: next post.
