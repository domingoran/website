Introduction

Every number in this series so far has measured retrieval, and every one of them has keyed correctness on a single field: chunk_id. A question counts as "answered" when the retrieved chunk IDs intersect the labelled relevant_chunk_ids — that is what Recall@K and MRR actually score. The previous post ended by naming the problem with that: it is a proxy for the thing we care about — whether the model produced the right answer — and the proxy is wrong in both directions. A retrieved chunk that isn't in the gold set can still contain everything the model needs, so the answer comes out right while Recall@K scores a miss. And the reverse: retrieval can be perfect and the model can still hallucinate or fumble the answer, a pure generation failure that is completely invisible to retrieval metrics.

This post closes that gap. Instead of comparing chunk IDs, it reads each generated answer against the reference answer and scores it directly, with a second LLM acting as an impartial judge. That lets us finally separate retrieval failures from generation failures — and, as it turns out, both of the proxy's blind spots show up clearly once you stop counting IDs and start reading answers.

Everything here runs locally, like the rest of the pipeline: the generator is llama3.1 via Ollama, the judge is a different model, qwen3, and the whole thing is one CLI flag — python main.py --eval-judge.

The proxy problem, restated

Retrieval metrics answer a narrow question: did the right passage make it into the context window? That is necessary but not sufficient. Between "the right passage was retrieved" and "the user got a correct answer" sits the entire generation step — the model still has to read the passages, pick the relevant facts, and write them down without inventing anything. Recall@K sees none of that.

Worse, it is miscalibrated at both ends:

  • Too pessimistic. Ground-truth relevant_chunk_ids label a specific passage as the answer, but a neighbouring chunk often carries the same fact. When the retriever returns that neighbour, Recall@K records a miss even though the model had everything it needed and answered correctly. We undercount our own success.
  • Too optimistic. Perfect retrieval guarantees nothing about the output. The model can misread the passage, attribute a fact to the wrong hormone, or answer a subtly different question — and Recall@K, which never looks at the generated text, calls it a win.
The only way to catch both is to grade the artefact the user actually receives: the answer. Doing that at scale by hand is slow and inconsistent, which is exactly the niche LLM-as-a-judge fills — a model reads the answer against a reference and scores it, cheaply and repeatably.

What a judge model does

The judge is a standalone evaluation stage. It runs after generation: --eval-score produces, for every question, the reference answer, the model's generated answer, and the list of chunk_ids that were retrieved. The judge (evaluation/eval_judge.py) loads that file and, for each row, builds a single scoring prompt and asks the judge model to return a strict JSON verdict — no prose, no markdown.

One detail matters for two of the metrics. To decide whether the answer is grounded (and whether its citations point somewhere real), the judge needs the actual passage text, not just the IDs. So before scoring, it fetches the retrieved chunks back out of Milvus by ID and renders them into the prompt as the retrieved context, each block tagged with its paper_id, section, and page: 

QUESTION:            <the question>
REFERENCE ANSWER:    <ground truth>
RETRIEVED CONTEXT:   [1] paper_id: ..., Section: ..., Page: ...
                     <chunk text>
                     [2] ...
GENERATED ANSWER:    <evaluate this>
The judge returns one integer (1–5) per metric plus a one-sentence rationale, and the harness clamps each score into range, tolerates a bare number where a model forgets the object wrapper, and strips any <think>…</think> reasoning block before parsing the JSON. A single bad or unparseable response never kills the run — that row is flagged with a judge_error and the aggregate is computed over whatever scored cleanly. Output lands in data/eval_judge_results.json.

Guarding against a biased judge

The obvious failure mode of LLM-as-a-judge is self-preference bias: a model asked to grade its own output tends to like what it sees. The pipeline avoids this structurally — the judge model (EVAL_JUDGE_MODEL, here qwen3) must differ from the generator model (OLLAMA_CHAT_MODEL, here llama3.1). If the two are ever set to the same value the harness logs a warning, because a judge scoring its own answers is measuring stylistic familiarity as much as correctness.

The prompt is written to be strict and reference-anchored rather than impressionistic. It tells the judge to reward a semantically correct answer even when it is longer or worded differently than the reference, and to penalise contradictions, wrong facts, and any claim not grounded in the supplied context. The goal is a judge that rewards being right, not sounding fluent — the two are easy to conflate, and conflating them is how you end up with an evaluator that rubber-stamps confident nonsense.

The five metrics

Each answer is scored on five axes, every one an integer from 1 (worst) to 5 (best). They are deliberately separated so a single "quality" number can't hide which part of the system failed:

  • Correctness — does the answer factually agree with the reference? This is the headline number: 5 = fully correct, 1 = wrong.
  • Completeness — does it cover all the key facts in the reference, or only some? An answer can be perfectly correct and still incomplete.
  • Answer relevance — does it actually address the question asked, without being evasive, off-topic, or padded?
  • Faithfulness — is every claim supported by the retrieved context? This is the hallucination detector: unsupported statements are penalised.
  • Citation quality — are the [paper_id, Section, Page] citations present, well-formed, and pointing to passages that actually support the claim?
The split is the whole point. Correctness and completeness grade the answer against the ground truth; faithfulness and citation quality grade it against the retrieved context; and answer relevance grades it against the question. Read together, they tell you not just whether an answer was good but where it broke.

Evaluation results

The judge scored 50 generated answers (17 easy / 17 medium / 16 hard) from the same three-book corpus used throughout the series, with qwen3 judging llama3.1's output. Scores are means on the 1–5 scale.

This is a clean, same-dataset comparison. The answer scores below and the retrieval scores they are lined up against come from the identical 50 questions — the very same run that produced the answers is the one whose retrieval was scored (MRR 0.90, Recall@1 0.74, Recall@3 0.88, Recall@5 0.89). So when correctness is set against Recall@K it is item-for-item, not directional: every answer's grade and its retrieval outcome describe the same question.

Overall 4.64 Mean of the five axes

Two numbers frame everything else. Answer relevance is a perfect 5.00 — across all 50 questions the model never once lost a mark for addressing the wrong thing. Read that one with suspicion, though: a flat 5.00 with zero variance is at least as likely to mean the judge doesn't resolve this axis finely — a small local model clamping everything on-topic to full marks — as it is to mean 50 genuinely flawless answers (a caveat I return to below). Either way the practical reading holds: whatever goes wrong from here is not a "didn't answer the question" problem; it is always a facts problem. And completeness (4.10) is the clear floor, sitting below correctness (4.70). The model gets the gist right but routinely leaves out secondary facts — the dominant weakness of this system is coverage, not accuracy.

This time the gradient is not a clean slide from easy to hard — the low point sits in the middle. Medium questions are the weakest tier (overall 4.58), scoring below both easy (4.69) and hard (4.64), and they drag down almost every axis at once: the lowest correctness (4.65), the lowest completeness (3.94), and the lowest faithfulness (4.71). Completeness remains the weakest axis everywhere (4.24 easy, 3.94 medium, 4.13 hard), and relevance never moves at all. Hard questions, strikingly, post the highest correctness of the three (4.75).

The reason the dip lands on medium rather than hard lines up with retrieval. On the very same run, medium had by far the weakest retrieval — Recall@5 = 0.79, against 0.94 on both easy and hard — so on those questions the model is more often reasoning without the gold passage in front of it: retrieval, not generation, is the limiter. Hard questions are the mirror image: strong retrieval (Recall@5 = 0.94, Recall@1 = 0.84) now paired with the strongest answers, the model exploiting the passages it was handed. Being able to say "medium is retrieval-limited, hard is not" is exactly the distinction retrieval metrics alone can never draw, and the reason a judge earns its keep.

The proxy was wrong both ways

This is the payoff. Because retrieval and answer quality were scored on the same 50 questions, the two blind spots Part 4 predicted stop being hypothetical and become countable — cross-tabulate each question's chunk-ID recall against the judge's correctness score and both directions fall out directly.

  • Retrieval metrics were too pessimistic — 2 questions. Two questions had Recall@5 = 0 — not a single gold chunk retrieved — yet the judge scored the generated answer a perfect 5/5 on correctness ("How do plant-based diets influence the gut microbiome?" and "What should be done instead of taking more vitamin D…?"). A neighbouring, non-gold chunk carried the needed fact; the model used it and answered correctly, while the chunk-ID metric recorded a clean miss. Two answers the retrieval proxy discarded as failures were, in fact, flawless.
  • And too optimistic — 2 questions. The mirror image: two questions retrieved a gold chunk at rank 1 (Recall@1 = 1) yet scored just 2/5 on correctness — the right passage was in context, faithfulness stayed reasonable, and the answer was still wrong. Asked what major challenge the author faced talking to audiences, the model answered about the lack of a proven causal link (correctness 2) where the reference was about patients recognising their own symptoms in the book's stories. Asked how researchers controlled for participants' romantic relationships in the clothing-choice study, it invented procedures the study never used (correctness 2). Both are pure generation failures, and both are completely invisible to Recall@K and MRR.

That is the entire case for the judge in two counts: 2 answers the retrieval proxy scored too harshly and 2 it scored too kindly. Retrieval was quietly better than Recall@K claimed on the ones it undercounted, and quietly worse on the ones where generation dropped the ball. Only by reading the generated text can you tell the two apart — and only then do you know whether your next hour is better spent on the retriever or on the generator.

What the judge can't tell you

An LLM judge is a measurement instrument, and like any instrument it has error. The scores above are useful precisely because they are separated by axis and anchored to a reference answer — but a few caveats keep them honest:

  • A small local judge scores generously — read the absolute numbers, not just the answers, with suspicion. qwen3 is itself a small model running on the same local hardware as everything else, and small judges are known to compress their scores toward the top of the scale: they discriminate less finely than a frontier judge or a panel of humans and rarely reach for the low end. The clearest symptom is sitting in the results table — answer relevance is a flat 5.00 across all 50 questions, with no variance at all. That is far more plausibly a judge ceiling — the axis the model resolves least well, so every on-topic answer collapses to full marks — than fifty perfect answers. The lesson generalises: treat the high absolutes (the 4.5–4.7 band) as somewhat inflated by a lenient judge, and trust the gaps — between axes, between difficulty tiers — far more than the raw altitude. A bigger or better-calibrated judge would very likely spread these scores out and pull the ceiling down.
  • The judge is itself a model. qwen3 can misread a subtle answer just as llama3.1 can. Using a different model from the generator removes self-preference bias, not judgement error. A rationale field is stored next to every score precisely so a human can spot a judge that has gone astray.
  • The ground truth is only as good as its reference answers. If a reference is itself incomplete, a fully correct generated answer can be marked down on completeness for including facts the reference simply omitted. Some of the completeness gap is real; some is reference noise.
  • One judge is one opinion. A single model gives a single verdict. A more robust setup would poll several judges, or calibrate the judge against a sample of human ratings, before trusting the absolute numbers rather than the relative trends.
None of this undoes the two findings — the correctness-vs-recall gap and the completeness cliff on hard questions are far larger than plausible judge noise. But it does mean the right way to read these scores is as a compass, not a ruler: they point clearly at where the system is weakest, which is all an evaluation harness needs to do.

Conclusion

For four posts the whole series graded itself on chunk_id overlap — a proxy that, it turns out, was wrong in both directions at once. Reading the answers directly resolves the ambiguity. The model rarely hallucinates (faithfulness 4.76) and is usually correct (4.70) — meaningfully more often than chunk-ID recall implied, which is the proxy being too harsh — while relevance pins the ceiling (5.0), as much a sign the small local judge doesn't resolve that axis as of flawless answers. But correctness short of 5, with grounded-yet-wrong answers, is the proxy being too generous: real generation failures that no retrieval metric could ever surface.

The single clearest signal is that completeness is the bottleneck (4.10 overall, and just 3.94 on medium questions) — and, crucially, the judge shows where to aim the fix. Medium is the weak tier on both axes at once: the weakest retrieval (Recall@5 = 0.79) and the weakest answers, so there the first lever is retrieval breadth — get the gold passage in front of the model more often. Hard questions are the opposite, pairing strong retrieval (Recall@5 = 0.94) with the highest correctness (4.75): the model exploits what it is handed, so the residual there is generation polish — prompting it to exhaust the context rather than skim it. Splitting the diagnosis this way is only possible because we scored the answers directly — chunk-ID recall alone could not tell a retrieval-limited tier from a generation-limited one. Either way, there is now a metric — answer completeness — that will move if the fix works.

The broader lesson: measure the artefact your user receives. Retrieval metrics are cheap, fast, and indispensable for tuning the retriever — but they are a means, not the end. The end is a correct answer, and the only way to know you produced one is to read it.
